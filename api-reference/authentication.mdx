## Overview

All Ref API requests require authentication using API keys passed as Bearer tokens in the Authorization header.

## Getting Your API Key

1. **Sign up** at [ref.tools](https://ref.tools)
2. **Navigate** to your dashboard
3. **Copy** your API key from the API Keys section
4. **Store securely** - never commit to version control

<Warning>
Keep your API key secure and never share it publicly. Treat it like a password.
</Warning>

## Authentication Methods

### Bearer Token (Recommended)

Include your API key in the Authorization header:

```bash
curl -X POST https://ref.tools/api/search \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"query": "React hooks"}'
```

### Environment Variables

Store your API key in environment variables:

<CodeGroup>

```bash Environment
export REF_API_KEY="your-api-key-here"
```

```javascript Node.js
const apiKey = process.env.REF_API_KEY;

const response = await fetch('https://ref.tools/api/search', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${apiKey}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({query: 'React hooks'})
});
```

```python Python
import os
import requests

api_key = os.environ['REF_API_KEY']

response = requests.post(
    'https://ref.tools/api/search',
    headers={
        'Authorization': f'Bearer {api_key}',
        'Content-Type': 'application/json'
    },
    json={'query': 'React hooks'}
)
```

</CodeGroup>

## API Key Management

### Key Properties

- **Format**: `ref_api_` followed by alphanumeric characters
- **Scope**: Full access to your account and team resources
- **Expiration**: Keys don't expire but can be rotated
- **Rate Limits**: Applied per API key

### Best Practices

<CardGroup cols={2}>
  <Card title="Security">
    - Store in environment variables
    - Never commit to version control
    - Use different keys per environment
    - Rotate keys regularly
  </Card>
  
  <Card title="Usage">
    - Use one key per application
    - Monitor usage in dashboard
    - Set up alerts for unusual activity
    - Keep backup keys ready
  </Card>
</CardGroup>

### Key Rotation

To rotate your API key:

1. Generate a new key in the dashboard
2. Update your applications with the new key
3. Test that everything works
4. Delete the old key

<Info>
You can have multiple active API keys to enable zero-downtime rotation.
</Info>

## Team Authentication

Team members can use:
- **Individual keys**: Personal API keys with team access
- **Team keys**: Shared keys for team resources
- **Service keys**: Dedicated keys for applications

Team API keys provide access to:
- Shared team indices
- Team credit pool
- Team-specific features

## Authentication Errors

### Common Error Responses

<AccordionGroup>
  <Accordion title="401 Unauthorized">
    ```json
    {
      "error": "unauthorized",
      "message": "Invalid API key",
      "code": "INVALID_API_KEY"
    }
    ```
    
    **Causes:**
    - Incorrect API key
    - Missing Authorization header
    - Malformed Bearer token
  </Accordion>

  <Accordion title="403 Forbidden">
    ```json
    {
      "error": "forbidden",
      "message": "Insufficient permissions",
      "code": "INSUFFICIENT_PERMISSIONS"
    }
    ```
    
    **Causes:**
    - Key doesn't have required permissions
    - Trying to access team resources without team access
    - Account suspended or restricted
  </Accordion>

  <Accordion title="429 Too Many Requests">
    ```json
    {
      "error": "rate_limit_exceeded",
      "message": "Too many requests",
      "retryAfter": 60
    }
    ```
    
    **Causes:**
    - Exceeded rate limits for your plan
    - Too many requests in short time period
    - Need to implement exponential backoff
  </Accordion>
</AccordionGroup>

## Testing Authentication

### Quick Test

Test your API key with a simple request:

```bash
curl -X POST https://ref.tools/api/search \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"query": "test"}' \
  -w "Status: %{http_code}\n"
```

Expected response: `Status: 200`

### Validation Script

<CodeGroup>

```javascript Node.js
async function validateApiKey(apiKey) {
  try {
    const response = await fetch('https://ref.tools/api/search', {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${apiKey}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({query: 'test'})
    });
    
    if (response.ok) {
      console.log('API key is valid');
      return true;
    } else {
      console.log('API key is invalid:', response.status);
      return false;
    }
  } catch (error) {
    console.error('Error validating API key:', error);
    return false;
  }
}
```

```python Python
import requests

def validate_api_key(api_key):
    try:
        response = requests.post(
            'https://ref.tools/api/search',
            headers={
                'Authorization': f'Bearer {api_key}',
                'Content-Type': 'application/json'
            },
            json={'query': 'test'}
        )
        
        if response.status_code == 200:
            print('API key is valid')
            return True
        else:
            print(f'API key is invalid: {response.status_code}')
            return False
    except Exception as error:
        print(f'Error validating API key: {error}')
        return False
```

</CodeGroup>

## Security Considerations

### Infrastructure Security

- **TLS 1.3**: All API communication uses TLS 1.3
- **HTTPS Only**: HTTP requests are redirected to HTTPS
- **No Logging**: API keys are never logged in plaintext
- **Encryption**: Keys stored encrypted at rest

### Application Security

- **Input Validation**: Always validate API responses
- **Error Handling**: Don't expose keys in error messages  
- **Network Security**: Use HTTPS in production
- **Access Control**: Limit key access to necessary personnel
