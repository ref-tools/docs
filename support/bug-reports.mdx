## Before Reporting

### Check First

Before creating a new bug report:

<AccordionGroup>
  <Accordion title="Search Existing Issues">
    Check if someone has already reported the same bug:
    - [Open Issues](https://github.com/ref-tools/ref/issues)
    - [Closed Issues](https://github.com/ref-tools/ref/issues?q=is%3Aissue+is%3Aclosed)
    - Use relevant keywords in your search
  </Accordion>

  <Accordion title="Check Service Status">
    Verify if the issue is due to a service outage:
    - [Status Page](https://status.ref.tools)
    - Check for ongoing incidents or maintenance
    - Look for recent status updates
  </Accordion>

  <Accordion title="Try Basic Troubleshooting">
    Common fixes that often resolve issues:
    - Verify your API key is correct
    - Check your internet connection
    - Restart the application/tool
    - Clear browser cache (for web interfaces)
    - Update to the latest version
  </Accordion>

  <Accordion title="Review Documentation">
    Make sure the behavior you're seeing is actually a bug:
    - [Feature Documentation](/features/search)
    - [Installation Guides](/installation/overview)
    - [API Reference](/api-reference/introduction)
  </Accordion>
</AccordionGroup>

## Creating a Bug Report

### Choose the Right Channel

**GitHub Issues** (Preferred for technical issues):
- API bugs and errors
- Client integration problems
- MCP server issues
- Documentation search problems
- Performance issues

**Email Support** (For sensitive or account issues):
- Billing and payment problems
- Account access issues
- Security-related concerns
- Private data or configuration issues

### Bug Report Template

Use this template for comprehensive bug reports:

```markdown
## Bug Description
Brief description of the issue

## Steps to Reproduce
1. Step one
2. Step two
3. Step three
4. See error

## Expected Behavior
What you expected to happen

## Actual Behavior
What actually happened

## Environment
- **Client/Tool**: (Claude Desktop, Cursor, VS Code, API, etc.)
- **Version**: (if known)
- **Operating System**: (macOS, Windows, Linux)
- **Browser**: (if applicable)
- **Node.js Version**: (for MCP servers)

## Additional Context
- Error messages (exact text)
- Screenshots or recordings
- Configuration files (remove sensitive data)
- API responses (if relevant)
```

## Common Bug Categories

### API Issues

<AccordionGroup>
  <Accordion title="Authentication Errors">
    **Symptoms**: "Invalid API key", "Unauthorized", "Forbidden" errors
    
    **Information to include**:
    - API key format (first/last 4 characters only)
    - Endpoint you're calling
    - Full error response
    - When the key was generated
  </Accordion>

  <Accordion title="Search Results Issues">
    **Symptoms**: No results, irrelevant results, or error responses
    
    **Information to include**:
    - Exact search query
    - Expected vs actual results
    - Account tier and available credits
    - Any specific documentation sources you expected
  </Accordion>

  <Accordion title="URL Reading Failures">
    **Symptoms**: Can't read specific URLs, timeout errors, or content parsing issues
    
    **Information to include**:
    - Exact URL that failed
    - Error message received
    - Whether the URL is publicly accessible
    - Expected content type (HTML, PDF, etc.)
  </Accordion>

  <Accordion title="Rate Limiting Issues">
    **Symptoms**: "Too many requests" errors when within expected limits
    
    **Information to include**:
    - Request frequency and timing
    - Account plan and limits
    - Error response headers
    - Time of occurrence
  </Accordion>
</AccordionGroup>

### Client Integration Issues

<AccordionGroup>
  <Accordion title="MCP Server Problems">
    **Symptoms**: Server won't start, connection errors, or missing tools
    
    **Information to include**:
    - MCP configuration file (remove API key)
    - Client application and version
    - Node.js version
    - Error logs from the client
    - Installation method used
  </Accordion>

  <Accordion title="Claude Desktop Integration">
    **Symptoms**: Tools not appearing or not working in Claude Desktop
    
    **Information to include**:
    - Claude Desktop version
    - MCP configuration file
    - Console/debug output
    - Operating system
  </Accordion>

  <Accordion title="Cursor Integration">
    **Symptoms**: Deeplink installation issues or missing functionality
    
    **Information to include**:
    - Cursor version
    - Installation method (deeplink, manual)
    - Error messages during installation
    - Configuration details
  </Accordion>

  <Accordion title="VS Code Issues">
    **Symptoms**: Extension not working or MCP server connection failures
    
    **Information to include**:
    - VS Code version
    - Extension name and version
    - Extension configuration
    - VS Code developer console output
  </Accordion>
</AccordionGroup>

## Information to Include

### Essential Information

Always include these details:

<CardGroup cols={2}>
  <Card title="Environment Details">
    - Operating system and version
    - Client application and version
    - Node.js version (for MCP)
    - Browser type and version (if applicable)
  </Card>
  
  <Card title="Error Information">
    - Exact error messages
    - Error codes or status codes
    - Stack traces (if available)
    - Console output
  </Card>
  
  <Card title="Reproduction Steps">
    - Clear, numbered steps
    - Minimal test case
    - When the issue started occurring
    - Frequency of the issue
  </Card>
  
  <Card title="Configuration">
    - Relevant config files (sanitized)
    - Environment variables (non-sensitive)
    - API endpoints being called
    - Request/response examples
  </Card>
</CardGroup>

### Helpful Extras

<AccordionGroup>
  <Accordion title="Screenshots">
    Visual evidence helps diagnose issues:
    - Error messages and dialogs
    - Unexpected UI behavior
    - Configuration screens
    - Console output
  </Accordion>

  <Accordion title="Screen Recordings">
    For complex issues, short videos can be very helpful:
    - Steps to reproduce the issue
    - Expected vs actual behavior
    - Keep recordings under 2 minutes
    - Use tools like Loom, CloudApp, or built-in screen recording
  </Accordion>

  <Accordion title="Log Files">
    Include relevant log output:
    - Application logs
    - Network request logs
    - Debug output
    - Remove sensitive information (API keys, personal data)
  </Accordion>

  <Accordion title="Network Information">
    For connectivity issues:
    - Network configuration (proxy, VPN)
    - Firewall settings
    - Geographic location
    - ISP information
  </Accordion>
</AccordionGroup>

## Security Issues

### Responsible Disclosure

For security vulnerabilities:

1. **DO NOT** create a public GitHub issue
2. **Email**: [security@ref.tools](mailto:security@ref.tools)
3. **Include**: Detailed vulnerability description
4. **Provide**: Steps to reproduce (if safe)
5. **Wait**: For our response before public disclosure

We commit to:
- Acknowledge receipt within 24 hours
- Provide initial assessment within 72 hours
- Keep you updated on fix progress
- Credit you for the discovery (if desired)

### Bug Bounty

We offer rewards for qualifying security issues:
- **Critical**: $500-2000
- **High**: $250-500
- **Medium**: $100-250
- **Low**: $50-100

## Privacy Considerations

When reporting bugs:

<Warning>
**Never include**:
- Full API keys (only first/last 4 characters)
- Personal information or private data
- Authentication tokens
- Customer data or private documentation content
- Internal URLs or system details
</Warning>

<Tip>
**Safe to include**:
- Sanitized configuration files
- Error messages and codes
- Public URLs and documentation
- System specifications and versions
</Tip>

## Bug Triage Process

### Our Response Process

1. **Acknowledgment**: We confirm receipt within 24-48 hours
2. **Initial Triage**: We categorize and prioritize the issue
3. **Investigation**: Our team reproduces and analyzes the bug
4. **Status Updates**: Regular updates on progress
5. **Resolution**: Fix deployed and issue closed
6. **Follow-up**: Verification that the fix resolves the issue

### Priority Levels

| Priority | Response Time | Description |
|----------|---------------|-------------|
| **Critical** | 2-4 hours | Service down, data loss, security issues |
| **High** | 1-2 business days | Major functionality broken |
| **Medium** | 3-5 business days | Important features affected |
| **Low** | 1-2 weeks | Minor issues, cosmetic problems |

## After Reporting

### What Happens Next

1. **Auto-Response**: You'll receive an automatic confirmation
2. **Triage**: We'll assign the appropriate priority and team member
3. **Updates**: We'll comment on progress and ask for clarification if needed
4. **Testing**: We may ask you to test fixes or provide additional information
5. **Resolution**: We'll notify you when the issue is fixed

## Quality Bug Reports

### Good Example

```
**Bug**: Search API returns 500 error for queries over 200 characters

**Steps to Reproduce**:
1. Send POST request to https://ref.tools/api/search
2. Include query with 201+ characters in request body
3. Observe 500 Internal Server Error response

**Expected**: Either successful search or proper error message
**Actual**: 500 error with no helpful error message

**Environment**:
- API called from Node.js 18.17.0
- Using curl for testing
- Account: Basic plan

**Error Response**:
```json
{
  "error": "internal_server_error",
  "message": "An unexpected error occurred"
}
```

**Additional Context**: 
Works fine with queries under 200 characters. Started happening yesterday.
```

### What Makes It Good

- Clear, specific title
- Reproducible steps
- Expected vs actual behavior
- Environment details
- Exact error messages
- Additional context
